I’m currently reading through Douglas Crockford’s “Javascript: The Good Parts” and one of the most useful things I’ve seen in that book is his explanation of how the this keyword is used in different circumstances. The following may have excerpts from the book that I may not have quoted. Except for the code, that’s all mine :)

Primarily, the this parameter’s value is determined by the function’s invocation pattern. What is an invocation pattern you might ask? An invocation pattern is how the function is called or invoked. There are four different invocation patterns that affect the this keyword:

  1. The Method Invocation Pattern

    • A Method is a function that is stored as a property of an object.
    • When a method is invoked, this is bound to that object.
    • The binding of this happens at invocation time (late binding).

  2. The Function Invocation Pattern

    • When a function is not the property of an object, then it is invoked as a function.
    • this is bound to the global object (weird but kind of makes sense).

Absolute Positioning in a Relatively Positioned Div

Jan 9th, 2014

This is one of those reminder posts.

Absolute positioning is great! It lets you position elements anywhere in the document with respect to the document body, even on top of other elements. That can be useful in some scenarios, but what I often find myself wanting to do is to absolutely position elements not with respect to the document body, but with respect to it’s containing element. For some reason I always forget how.

When an absolutely positioned element (an element with CSS style position: absolute) is given a left, right, top or bottom value, it is positioned with respect to the top left corner of the document, even when it is nested inside other elements. This is always the case EXCEPT when its direct parent element has position: relative. In this situation, the absolute positioning (top, left, right and bottom) is done with respect to the top left corner of this parent element! Also, properties that reference percentages are also now taken with respect to the relatively positioned parent element, so width: 50% is not 50% the width of the entire document, but 50% of the parent element. Good to know!

Leave me a comment if I’m mistaken with anything I’ve said in this post.

Changing the Octopress Code Theme to Something More Sublime

Jan 4th, 2014

One of the first things I noticed when looking into using Octopress was the default theme used for code examples. I’m not usually hugely picky about the color scheme of my code, but I like it to at least match its surroundings and be as easy on the eyes as possible. Vanilla Octopress comes with two different themes, a light theme and a dark theme:

Light

Dark

As you can probably tell, neither of those suit the design of my blog (or at least, not as well as I’d like at the time of this writing) and if you’re reading this article then chances are it doesn’t suit your design either. Hopefully this post can help.

This might not be the best tutorial (it’s pretty quick and dirty), but it will give you complete control over the styling of your octopress code blocks and give you a better sense of how things fit together.

My editor of choice is the beautiful Sumblime Text editor that looks a little something like this:

Wouldn’t it be nice to have that style here? As far as I know, there are no drag-and-drop solutions to code theming in octopress, so it’s time to dive right into the style-sheets themselves and do some asploring!

Octopress uses SASS CSS preprocessing to generate its style-sheets. For more information on SASS check here. I highly recommend using it since without it, extensive style-sheets can become a horrible repeating steaming file of mess and disarray.

Changing the Code Font:

The first thing I wanted to change was the font used in the code blocks. Sublime Text uses “Consolas” as its default font but I want to use something that matches my blog’s font. I decided on using the Ubuntu Mono font for this example.

I really like Google Fonts. You can find whatever font you want to use from there and include it in your html with the code provided by google. It’s like a CDN for fonts! I ended up importing the font through a link html tag in the head, but you can also do this through an @import statement in CSS. To change the head tag of your blog, you can edit the code found in source_includes\custom\head.html and add the link tag to the font there:

<link href=“http://fonts.googleapis.com/css?family=Ubuntu+Mono” rel=“stylesheet” type=“text/css”>

Now that we’ve imported the font from google, we need to tell SASS where to use it!

As you might expect, the font types are specified in the sass file called _typography.scss. This file can be found in sass\base_typography.scss

Now change the mono font variable (should be on line four-ish) to the new one you’ve picked. In my case my new mono font looks like this:

$mono: ‘Ubuntu Mono’, sans-serif !default;

Now in the sass files, whenever you reference the variable $mono, you’re referencing the imported google font. For more information on what that !default does check this out, but it pretty much does exactly what you’d think it’d do.

So now we have some shiny new font! Check it out!

Changing the Header:

Now how about we change that gross silver bar at the top of the block? Right now it’s using an image file found in source\images\code_bg.png, so lets go into the sass and tell it to drop that image like it’s hot.

Most code formatting style stuff is handled in the _syntax.scss file found in sass\partials_syntax.scss. You can change the background of the title bar on line 245. I’ve changed it from

css background: #aaaaaa image-url(“code_bg.png”) top repeat-x;
to
css background: #FAFAFA;

I’m a simple man and I like my font to be simple as well. I’ve also changed the font on line 244 from

css font-family:“Helvetica Neue”, Arial, “Lucida Grande”, “Lucida Sans Unicode”, Lucida, sans-serif;
to
css font-family: $mono;

I also got rid of the text shadow on line 239. Not my style.

So now we got something that looks like this:

Getting closer! Now let’s get into the fun stuff:

Changing Colors:

All code colors are stored in sass\base\_solarized.scss as variables. The variables that have a 0 in front of the postfix are the colors used for the “dark” theme and the others for the light theme. There’s a conditional statement that controls the theme that inverts the colors based on whether or not you want a dark theme or light theme. Dont worry about that all that too much (unless you plan on using both themes) and use the comments as a point of reference.

These color variables are used mostly in sass\partials\_syntax.css for code syntax highlighting.

The $solarized variable on line 18 is what lets you choose between using the dark and light themes like those pictured at the start of this post.

To get that sublime feel, I took a print-screen of a page of code in my sublime editor and used the eyedropper tool in GIMP 2.0 to get the hex values of the colors. You can change them to whatever colors you want though. Let me walk you through that process a bit.

Background Colors:

The code background color is the first line in _solarized.scss with the comment ‘darkest blue’. I set this to sublime black:

line 1: $base03: #272822 !default;

The line numbers background (that strip on the left of the code) is $base02. I decided to choose the same light grey as the selected line in sublime text which came out to this:

line 2: $base02: #3e3d32 !default;

Line Numbers:

Again, taking my trusty eyedropper, I scraped the line number text color from sublime and inserted it into line 20 of _syntax.scss to make all spans of the line-numbers class have that soft shade of white.

line 20: span{color:#8f908a !important;}

Ok! so now we’ve changed some colors, let’s take a gander:

Now there’s only one thing left to do…

Code Syntax Highlighting Colors:

You can change it to whatever you want based on the short classes in _syntax.scss.

Actual syntax highlighting is done with the python plugin pygments.py during the generate process so if you want to really customize your code syntax highlighting, look into it. As far as I can understand, it works in concert with Liquid, the templating engine octopress uses to wrap the code input in differently classed span tags that get their colors from the syntax.scss file. By changing the color variables in the solarized.scss file and using those variables to change the colors in those tiny pygment span classes in _syntax.css (roughly lines 128-193) then you can customize the syntax highlighting colors. Again, I used my eyedropper method to rip the colors from sublime and change the color variables with those ripped colors.

A bit of a warning: Changing the colors of the pygment classes may look great in one language but horrible in another. This is because pygements reuses classes between languages and sometimes you get code that has classes who’s colors don’t play nice. Change this around at your own risk, but with enough fiddling, I think it can look great for any language. So here’s the finished product:

Very sublimey, and definitely better than the default at least. So go nuts and make your own custom code theme!

Boutysource: A Bountiful Force for Open Source

Dec 16th, 2013

Bountysource is an open source project that allows users (or anybody really) that wants to see a feature or bug-fix implemented in an open source project post a bounty for it. What is a bounty? Anybody who’s a member of Stack Overflow (or watched Dog the Bounty Hunter) knows that a bounty is a sum of money offered in return for the solution of a problem. Sounds like the service industry right? Except instead of trusting the solution to professionals, it’s left open to the general public. Anyone who thinks they can provide a solution can try, but only one person will have their solution accepted and will be paid.

An example: Company A is working on a project that uses an open source tool that hasn’t been maintained for a while because the people who maintain it have mostly stopped development on the project due to a lack of interest/funding. Company A would really like to see feature X in the tool. Otherwise, they’ll have to remove the tool from their project and start again from scratch with something else that has that feature. The problem is that that takes time and money and it might not suit all their other needs. In swoops Boutnysource in all it’s greenish glory! Company A posts a bounty of Y dollars on Bountysource. John the part-time developer sees this bounty, knows he could add the feature in a reasonable amount of time and sets to work on it. A short time later, the feature is implemented, John gets paid and Company A has saved lots of time and money. That’s just one idyllic scenario that makes Bountysource so attractive.

Esentially, Bountysource is to open source development what Kickstarter is to other projects, with one important twist; rather than having the people pledge money to companies for new software features, they pay the people to develop new software features. Sounds neat right? They’re changing the Kickstarter formula by flipping it on it’s head and asking for work in return for cash.

Of course, it would be ridiculous to ditch what made kickstarter so successful and so they also have a section for fundraisers that work with pledges, the way you’d expect.

How does it work?

An excerpt from the Bountysource FAQ:

How Bounties Work:

  1. Users find bounties on open issues or feature requests they want to see addressed.
  2. These users spread the word about the bounty, enticing developers to create a solution.
  3. Developers create solutions and claim the bounty on Bountysource.
  4. Backers can accept or reject the claim.
  5. If accepted, Bountysource pays the bounty to the developer.

How Fundraisers Work:

  1. Anyone can come to Bountysource and create a Fundraiser. Open-source fundraisers are typically used to raise money for new projects, big updates to existing projects, or to raise money for bounties.
  2. The Fundraiser creator spreads the word about the Fundraiser to the appropriate communities.
  3. Anyone can come to Bountysource and make pledges to a Fundraiser, helping it reach its funding goal in time.

It’s point number four of how bounties work that has me slightly wary. But that is a very small slightly. I’m sure you’ve heard stories of Kickstarter projects running out of funding halfway through development and leaving all those contributors penniless with nothing to show for it. I can see the same happening here where one project posts a large bounty, someone puts many weekends into it and are rewarded with a big fat ‘this did not meet our required specifications’ even though it does and then they never get paid.

From what I can tell, right now the service is mostly being used to power web development projects and other tools. It would be neat to see something like this applied to a gaming project. If you can get a talented and enthusiastic enough group of people to back a project with their work, I think some really creative and impressive things can come from this approach.

When I first read about Bountysource, I got all excited and warm and fuzzy inside. Maybe I’m naive, but this really does sound like an amazing idea, and with sites like GitHub around making open source project contribution a simple thing, I think Bountysource has a very promising future. Now pardon me while I go hunt some bounties :3
